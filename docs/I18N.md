# Internacionalização (M3.5)

Sistema leve de i18n baseado em dicionários, sem roteamento por locale (evita a
complexidade de segmentos `[locale]` no App Router do Next 16). O idioma é
escolhido pelo usuário e persistido em cookie.

## Como funciona

- **`src/lib/i18n.ts`** — módulo puro (sem env/IO; testável e seguro no client).
  - Locales: `pt-BR` (padrão), `en`, `es`.
  - `getDictionary(locale)` retorna o dicionário; `resolveLocale(valor)` mapeia
    cookie/`Accept-Language` para um locale suportado; `interpolate(tmpl, vars)`
    resolve `{placeholders}`.
- **Cookie `NEXT_LOCALE`** — definido por `src/components/locale-switcher.tsx`
  (recarrega a página; o server component relê o cookie e re-renderiza).
- **Server components** `/p/[slug]` e `/embed/[slug]` leem o cookie via
  `cookies()` e passam `locale` para `PublicForm`.

## Estado atual

- ✅ **Formulário público** (`PublicForm`) — totalmente traduzido (pt/en/es),
  com seletor de idioma. É a superfície externa (pacientes) que mais se beneficia
  de EN/ES.
- ⏳ **Painel admin** — permanece em pt-BR. A infraestrutura já está pronta; a
  migração das strings do admin é incremental (superfície grande e mecânica,
  prioridade menor — a equipe da clínica usa pt-BR).

## Como estender ao admin (incremental)

1. Adicionar as chaves ao `Dictionary` em `src/lib/i18n.ts` (ex.: `admin.nav.*`),
   replicando nos três locales (o teste `tests/i18n.test.ts` garante paridade de
   chaves entre locales).
2. Ler o locale no layout do admin (`cookies()` → `resolveLocale`) e passar o
   dicionário aos componentes.
3. Adicionar o `LocaleSwitcher` ao header do admin quando o painel estiver
   traduzido (evitar expor o seletor enquanto as strings ainda estão em pt-BR).

## Conteúdo do usuário vs. chrome

Apenas o **chrome** (rótulos da UI) é traduzido. Conteúdo autorado pelo admin
(títulos de pesquisa, perguntas, opções, política de privacidade, mensagem de
agradecimento) é exibido no idioma em que foi cadastrado — não é traduzido
automaticamente.
